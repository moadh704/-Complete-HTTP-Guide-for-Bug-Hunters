What this guide is for

HTTP is a plain-text protocol. Every field in a request or response is either doing its job correctly, or quietly telling you where the application's assumptions break down. Most web vulnerabilities are not exotic — they're a server trusting something it shouldn't, or a client sending something a server never expected. This guide walks through the protocol itself, then a live console to try it, then a reference of what to look for and where it's currently ranked in the industry's own data.

Before you test anything

Only probe applications you own, or that have explicitly authorized you through a bug bounty program or signed engagement scope. Sending crafted requests to systems without permission can be a criminal offense in most jurisdictions, regardless of intent. See Responsible testing for legal scope-of-work basics and legal practice targets.

The shape of a transaction

Four steps, four testing points

Every HTTP exchange — a page load, an API call, a webhook — moves through the same four beats. Each one is a distinct place to poke.

1

Client builds the request

A method, a path, headers, and — for most methods other than GET — a body get assembled by the browser, script, or tool making the call.

Every field here is attacker-controlled input
2

Server parses and authorizes

The server has to decide what the request means, who's making it, and whether they're allowed to. This is where access-control logic either holds or doesn't.

Change IDs, roles, and content types; watch what's actually checked
3

Server builds the response

Status code, headers, and a body get assembled — including whatever the application decided was safe to say, sometimes more than it meant to.

Error pages and headers leak stack traces, versions, and config detail
4

Client renders the response

HTML gets parsed, scripts get executed, JSON gets read by application code. Whatever came back is now trusted by default.

Unsanitized output here is how stored/reflected XSS happens
Anatomy · Request

What a request is actually made of

A request line, a set of headers, a blank line, and an optional body. That's the entire format — everything else is convention.

GET

Fetch a resource. Shouldn't change server state — but plenty of real apps break this rule, which is itself worth testing.

POST

Create a resource or trigger an action. Not idempotent — sending it twice can mean two effects.

PUT

Replace a resource entirely with what's sent. Idempotent by definition, though implementations vary.

DELETE

Remove the target resource. High-value target for missing authorization checks.

The four pieces

  • Request line — method, path, HTTP version
  • Headers — metadata about the request and client
  • Blank line — separates headers from body
  • Body — optional payload (form data, JSON, XML, binary)
Raw request over the wire
GET /api/users/123 HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjMifQ...
Cookie: session=af31c9; role=user

Every line above is something a client chose to send. Nothing in that list is verified until the server checks it — which is precisely the point.

Methods in depth

GET

Retrieve data; parameters usually ride in the URL.

Test for: IDOR via changed IDs, SSRF in URL-fetching parameters, sensitive data logged via query strings.

POST

Submit data or trigger a state change.

Test for: CSRF on state-changing actions, mass assignment via extra JSON fields, business-logic replay.

PUT

Full replacement of a resource.

Test for: missing ownership checks, mass assignment of fields the UI never exposes.

PATCH

Partial update to a resource.

Test for: the same mass-assignment risk as PUT, often with weaker validation since it's implemented later.

DELETE

Remove a resource.

Test for: authorization — can you delete something that isn't yours?

OPTIONS / HEAD / TRACE

Introspection methods.

Test for: OPTIONS reveals allowed methods; TRACE enabled on production is a red flag (Cross-Site Tracing) and should be disabled.

Headers worth reading twice

Authorization

Carries the credential — Basic, Bearer token, API key.

Look for tokens with no expiry, weak JWT signing, or tokens that leak into logs and Referer headers.

Cookie

Session and preference state sent by the browser automatically.

Check flags on the corresponding Set-Cookie: missing Secure, HttpOnly, or SameSite opens the door to theft or CSRF.

Content-Type

Tells the server how to parse the body.

Switching between json, form-urlencoded, and xml sometimes reaches an entirely different, less-validated code path.

X-Forwarded-For / X-Forwarded-Host

Meant to preserve client info behind a proxy.

Client-supplied and trivially spoofed — dangerous if used for IP allow-lists or password-reset link generation.

Origin

Where a cross-origin request claims to come from.

Servers that reflect this value back in Access-Control-Allow-Origin without an allow-list create a CORS hole.

Referer

The page a request was linked from.

Sometimes (wrongly) used as CSRF protection — worth checking whether it's actually validated server-side.

Body formats

Form-encoded

Example
username=admin&password=hunter2&role=user

Try HTTP Parameter Pollution — the same key twice — and see which value the server actually uses.

JSON

Example
{
  "username": "admin",
  "role": "user"
}

Add fields the API never documented ("role": "admin") — mass assignment happens when the server trusts the whole object.

XML

Example
<user>
  <username>admin</username>
  <role>user</role>
</user>

XML parsers that resolve external entities are vulnerable to XXE — potentially reading local files or reaching internal services.

Also test

Conflicting Content-Length and Transfer-Encoding headers between a front-end proxy and back-end server — the root cause of HTTP request smuggling. More in Advanced techniques.

Where parameters travel

URL / query string

GET /search?q=test&category=books&limit=10

Visible in browser history, server logs, and the Referer header sent to any linked third party — never a safe place for secrets.

Path segments

GET /api/users/123/orders/456

The classic IDOR surface — increment, decrement, or swap the ID and see whose data comes back.

Body parameters

POST /api/users
Content-Type: application/json

{"username":"admin"}

Where most injection testing happens — SQL, command, template, and NoSQL contexts all vary by field.

Header parameters

X-API-Key: 12345abcde

Custom headers are easy to forget when reviewing input validation — they're still attacker-controlled input.

Anatomy · Response

What the server sends back

Status line, headers, blank line, body — the mirror image of a request, and just as worth reading in full.

Raw response over the wire
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 47
Set-Cookie: session=af31c9; Secure; HttpOnly; SameSite=Lax
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000

{"status":"success","data":{"id":123}}

Read the headers before the body. Half of what a response gives away is metadata the developer never meant to expose.

Status codes, by class

100 Continue

Client should proceed with sending the body.

Rarely security-relevant, but confirms the server negotiates in stages — useful when profiling proxies.

101 Switching Protocols

Upgrading, typically to WebSocket.

Check whether the upgraded WebSocket channel re-applies the same auth as the REST API.

200 OK

Standard success.

Compare 200 vs. 403 timing/body across privilege levels — inconsistent handling often signals an access-control gap.

201 Created

A new resource now exists.

Confirm the created object actually enforces ownership — creation endpoints are often less guarded than reads.

202 Accepted

Queued for async processing.

Race conditions live here — the request "succeeded" before validation actually ran.

204 No Content

Success, empty body.

Still check headers — a 204 can leak just as much via Set-Cookie or CORS headers as a 200 with a body.

301 Moved Permanently

Resource lives at a new URL, cache it.

User-controlled redirect targets are the open-redirect testing surface.

302 Found

Temporary redirect.

Same open-redirect risk; also check whether auth headers are re-sent to the new host.

304 Not Modified

Cached copy is still valid.

Confirm cache keys include the auth context — shared caches have leaked private responses this way.

307 / 308 Redirect

Like 302/301, but the method and body must be preserved.

A POST that gets silently redirected cross-origin can leak a body an app author never expected to travel.

400 Bad Request

Malformed request.

Read the error text — frameworks often echo back internal validation logic or field names here.

401 Unauthorized

No valid credentials.

Check whether the message differs for "wrong password" vs. "unknown user" — that gap enables user enumeration.

403 Forbidden

Understood, but refused.

Try alternate methods, trailing slashes, or case changes on the path — inconsistent routing has bypassed 403s before.

404 Not Found

Nothing there.

A 404 on a private object versus a 403 on a private object you don't own can itself confirm the object exists.

405 Method Not Allowed

Wrong verb for this path.

Enumerate methods with OPTIONS to see what a route actually accepts beyond the documented one.

409 Conflict

Request conflicts with current state.

Good signal for race-condition hunting — see who gets the conflict when two requests land at once.

422 Unprocessable Entity

Understood, but semantically invalid.

Detailed validation errors here are a common source of internal schema disclosure.

429 Too Many Requests

Rate limit hit.

Confirm the limit is keyed on something you can't trivially rotate (IP alone is rarely enough).

500 Internal Server Error

Unhandled failure.

Stack traces here are a goldmine — framework version, file paths, sometimes query text.

502 Bad Gateway

Upstream server gave an invalid response.

Can indicate a request-smuggling attempt confused the proxy chain.

503 Service Unavailable

Temporarily overloaded or down.

Worth noting if it appears right after a load or race-condition test — a sign you found a resource-exhaustion path.

504 Gateway Timeout

Upstream didn't respond in time.

A timing oracle — different delays for valid vs. invalid input can leak information even with an identical status code.

Response headers that carry security weight

Set-Cookie

Missing Secure, HttpOnly, or SameSite opens the door to interception, script-read theft, or cross-site request forgery respectively.

Content-Security-Policy

The real modern defense against XSS. Look for unsafe-inline, unsafe-eval, or overly broad host allow-lists that weaken it back down.

Strict-Transport-Security

Forces HTTPS for future visits. Missing it allows a downgrade attack on the very first request.

X-Content-Type-Options

Should be nosniff. Without it, a browser may execute a file as a different type than declared.

Permissions-Policy

Restricts browser features (camera, geolocation, etc.) per origin. It replaced the older, now-deprecated Feature-Policy header.

Access-Control-Allow-Origin / -Credentials

The CORS pair. Reflecting an arbitrary Origin while Allow-Credentials is true lets any site read authenticated responses.

Worth knowing

X-Frame-Options still works, but Content-Security-Policy: frame-ancestors is the modern replacement and takes precedence in browsers that support both. X-XSS-Protection is deprecated and ignored by current major browsers — a CSP does that job now.

Body formats

HTML

Check rendered comments, inline scripts, and hidden form fields — data meant for developers sometimes ships to production.

JSON

The most common API format. Compare full responses across roles — over-fetching (returning fields the UI just hides) is extremely common.

XML / SOAP

Older systems especially. Same XXE risk as request bodies, mirrored on the way back.

Reference · Risk categories

                    
OWASP Top 10:2025, translated into HTTP terms

                    
OWASP published the first major revision of the Top 10 since 2021 in November 2025 — plenty of guides you'll find still teach the outdated ranking. Two categories are new, one was folded into another, and the order shifted meaningfully. This maps each current category to what it actually looks like on the wire.

                


                

                    
                

                

                    
                    
                    
                    
                    
                    
                    
                    
                    
                    
                    
                


                


                    

                        
A01
Broken Access Control

                        
The server fails to enforce what a given user is actually allowed to do or see. Holds the #1 spot for the fourth edition running, and in 2025 it absorbed Server-Side Request Forgery (SSRF), which used to be its own category.

                        
On the wire
Swap an object ID and get someone else's data back (BOLA); reach an admin route as a normal user (BFLA); a URL-fetch feature that can be pointed at internal infrastructure (SSRF).

                        
Testing approach
Replay the same request as a lower-privileged account; walk through object IDs sequentially; test any "fetch a URL" feature against internal-looking hosts, only within an authorized scope.

                        
Burp AutorizeAuth Analyzer

                    


                    

                        
A02
Security Misconfiguration

                        
The stack — server, framework, cloud storage, containers, security headers — is left in an insecure default or half-hardened state. The single biggest mover in 2025, climbing from #5 to #2.

                        
On the wire
Verbose stack traces on error, exposed admin panels, open cloud storage, missing security headers, default credentials still enabled.

                        
Testing approach
Request malformed input and nonexistent resources to see what leaks; check for directory listing and exposed .git/.env files; run a header scan.

                        
NucleiNikto

                    


                    

                        
A03
Software Supply Chain Failures

                        
New in 2025 — a broader replacement for 2021's "Vulnerable and Outdated Components." Covers compromised dependencies, malicious package updates, and weak build/distribution pipelines, not just an unpatched library.

                        
On the wire
A page pulling a tampered third-party script with no integrity check; a backend dependency with a known CVE still in production.

                        
Testing approach
Inventory client and server dependencies against known CVEs; check for Subresource Integrity on third-party <script> tags; review what a CI/CD pipeline trusts by default.

                        
npm auditpip-auditDependency-Track

                    


                    

                        
A04
Cryptographic Failures

                        
Sensitive data moves or rests without adequate protection — weak or absent TLS, home-grown crypto, predictable tokens, secrets sitting in plaintext.

                        
On the wire
HTTP still reachable alongside HTTPS, mixed content, session tokens with visible patterns, sensitive fields riding in a URL.

                        
Testing approach
Check for HTTP fallback and mixed content; inspect the TLS configuration; look at how session tokens are generated for predictability.

                        
testssl.shSSL Labs

                    


                    

                        
A05
Injection

                        
Untrusted input reaches an interpreter — SQL, shell, LDAP, a template engine, NoSQL — as part of a command instead of pure data. Ranked lower than in 2021, but still one of the most commonly exploited categories in practice.

                        
On the wire
A parameter that changes server behavior when you add a quote or boolean logic; a search field that leaks a database error verbatim.

                        
Testing approach
Supply syntactically meaningful characters for the context into every input and watch for behavior changes, timing differences, or verbose errors — inside an authorized scope only.

                        
sqlmapBurp Intruderffuf

                    


                    

                        
A06
Insecure Design

                        
The flaw lives in the architecture or business logic itself, not a single coding mistake — missing threat modeling, or trusting the client to enforce rules only the server should enforce.

                        
On the wire
Pricing or discount logic manipulated by replaying requests out of order; a checkout flow with no server-side re-validation of quantity or currency.

                        
Testing approach
Map the intended workflow, then try skipping steps, replaying requests out of sequence, or altering client-controlled values a well-designed server should re-check.

                        
Manual reviewBurp Repeater

                    


                    

                        
A07
Authentication Failures

                        
Weaknesses in login, session, and identity handling — the 2025 name for what was "Identification and Authentication Failures" in 2021. Holds steady at #7.

                        
On the wire
No rate limiting on login, session tokens that outlive logout, missing multi-factor on sensitive actions.

                        
Testing approach
Check for lockout/rate-limiting on auth endpoints; confirm sessions are invalidated after logout and password change; examine password-reset token generation and expiry.

                        
Burp Intruder

                    


                    

                        
A08
Software or Data Integrity Failures

                        
Code or data gets trusted without verifying it hasn't been tampered with — insecure deserialization, unsigned auto-updates, a deploy pipeline that accepts unsigned artifacts.

                        
On the wire
An endpoint that deserializes an object straight from client-supplied data; an update mechanism with no signature check.

                        
Testing approach
Identify serialization formats in transit; test whether a tampered serialized object is still accepted; check whether update/deploy channels verify signatures.

                        
Manual protocol analysis

                    


                    

                        
A09
Security Logging and Alerting Failures

                        
Renamed in 2025 from "Logging and Monitoring Failures" to make the point explicit: logs existing isn't the same as anyone acting on them.

                        
On the wire
Login failures, access-control denials, and admin actions that never generate a log entry or trigger an alert.

                        
Testing approach
During an authorized engagement, generate a deliberately suspicious event and check whether it surfaces anywhere the defenders would see it — this is more of a program-maturity question than a single exploit.

                        
Program review

                    


                    

                        
A10
Mishandling of Exceptional Conditions

                        
Brand new for 2025. Covers what happens when an app hits an error or edge case and fails open, leaks internals, or lands in an undefined state instead of failing closed.

                        
On the wire
A stack trace exposing environment variables; access granted when a downstream permission check errors out instead of denying; an unexpected parameter reaching an unhandled code path.

                        
Testing approach
Deliberately send malformed, oversized, or unexpected input and check whether the app fails safe or fails open; read every error response for anything it shouldn't say.

                        
Manual fuzzingBurp Suite

Beyond the basics

                    
Advanced techniques

                    
Once the fundamentals are automatic, these are the categories that separate a scanner report from an actual finding.

                

                

                    

                        
  HTTP Request Smuggling

                        
Front-end proxies and back-end servers sometimes disagree about where one request ends and the next begins — usually because Content-Length and Transfer-Encoding are both present and parsed differently by each hop. That disagreement lets one request "smuggle" a hidden second request past whatever the front end thought it was inspecting.

                        
HTTP Request Smuggler (Burp)

                    

                    

                        
  Race Conditions

                        
Servers that check-then-act (check your balance, then debit it) are vulnerable if enough identical requests land in the same instant — before the first one finishes writing its result. Single-packet attack techniques make this reliable even over normal internet latency.

                        
Turbo Intruder

                    

                    

                        
  JWT Attacks

                        
Tokens are only as trustworthy as the verification step. Watch for servers that read the algorithm from the token's own header instead of enforcing one server-side (an alg: none downgrade), weak or shared HMAC secrets, and kid header values that get fed straight into a file or key lookup.

                        
jwt_tool

                    

                    

                        
  GraphQL-specific risks

                        
Introspection left enabled in production hands over the entire schema. Aliasing and batching can bypass a rate limit designed around one query per request. Nested queries can also be used to force expensive, resource-exhausting joins.

                        
InQLGraphQL Voyager

                    

                    

                        
  WebSocket security

                        
A WebSocket upgrade often inherits none of the REST API's authorization checks by default. Confirm the Origin header is validated on the handshake, and that every message is still authorized individually, not just the initial connection.

                        
Burp WebSocket history

                    

                    

                        
  Business logic flaws

                        
No scanner finds these reliably — they require actually understanding what the application is supposed to do, then testing what happens when steps are skipped, reordered, or repeated. Coupon stacking, negative quantities, and currency-rounding abuse all live here.

Before you submit anything

                    
Responsible testing & disclosure

                    
Technique is only half the job. Scope, authorization, and how you report a finding are what separate a bug bounty payout from a criminal complaint — and they're not optional reading.

                


                

                    

                        
  Stay inside written scope

                        
A bug bounty program's scope document is a contract, not a suggestion — it lists exactly which domains, apps, and test types are authorized, and what's explicitly off-limits (often things like physical attacks, social engineering of staff, or denial-of-service testing).

                    

                    

                        
  Don't cause damage to prove a point

                        
Once you've confirmed a finding — a first successful SQL injection, an IDOR that returns another user's record — stop. Mass data extraction, account creation at scale, or destructive writes turn a valid finding into real harm, and most programs will disqualify or ban for it.

                    

                    

                        
  Follow coordinated disclosure timelines

                        
Give the vendor a reasonable window to fix a confirmed issue before any public write-up — typically 90 days is treated as an industry norm, though a program's own policy always takes precedence.

                    

                    

                        
  Know the legal baseline

                        
Testing systems without authorization can violate computer-crime law even when no damage occurs and the intent was good faith — the U.S. Computer Fraud and Abuse Act and similar statutes elsewhere don't require malicious intent to apply. This isn't legal advice; when in doubt, get explicit written permission first.

                    

                


                

                     Where authorized testing actually happens
                    
Bug bounty platforms formalize scope and safe harbor so you're never guessing: HackerOne, Bugcrowd, Intigriti, and YesWeHack are the largest. Read the program's policy in full before your first request.

                

            


            

                

                    
Knowledge check

                    
Test what stuck

                    
Seven questions, including a couple on what actually changed in the 2025 OWASP update.

                


                

                    

                        
1. Which of these methods is not expected to be idempotent?

                        

                            
                            
                            
                            
                        

                        
POST typically creates a resource or triggers an action each time it's called. GET, PUT, and DELETE are all defined to be idempotent — repeating them shouldn't change the outcome further, even though real implementations sometimes get this wrong.

                    


                    

                        
2. In the OWASP Top 10:2025, which category absorbed SSRF, previously its own standalone category?

                        

                            
                            
                            
                            
                        

                        
SSRF was folded into Broken Access Control in the 2025 edition, which also now explicitly covers BOLA and BFLA-style API authorization failures.

                    


                    

                        
3. Which status code indicates a resource was created successfully?

                        

                            
                            
                            
                            
                        

                        
201 specifically signals that a new resource now exists, often alongside a Location header pointing to it.

                    


                    

                        
4. Which header is the modern, most reliable way to prevent clickjacking?

                        

                            
                            
                            
                            
                        

                        
X-Frame-Options still works and is fine as a fallback, but CSP's frame-ancestors directive is the current standard and takes precedence where both are supported.

                    


                    

                        
5. Which CORS setup lets an attacker's site read a victim's authenticated API responses?

                        

                            
                            
                            
                            
                        

                        
Dynamically reflecting whatever Origin a request sends, combined with allowing credentials, effectively lets any site make authenticated requests on a logged-in victim's behalf and read the result.

                    


                    

                        
6. An API decides how to verify a JWT's signature based on the "alg" field inside the token itself. What's the risk?

                        

                            
                            
                            
                            
                        

                        
The server — not the token — must decide which algorithm is acceptable. Trusting the token's own header for that decision is how "alg: none" and algorithm-confusion attacks succeed.

                    


                    

                        
7. HTTP request smuggling exploits disagreement between a front-end and back-end server about...

                        

                            
                            
                            
                            
                        

                        
Usually caused by conflicting Content-Length and Transfer-Encoding headers being parsed differently by each hop, letting one request smuggle a hidden second one past the front end.
