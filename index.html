Introduction to HTTP

HTTP (Hypertext Transfer Protocol) is the foundation of data communication on the World Wide Web. Understanding HTTP requests and responses is crucial for bug hunters to identify security vulnerabilities in web applications.

Bug Hunter Tip: Most web vulnerabilities stem from improper handling of HTTP requests or insecure transmission of data. Mastering HTTP gives you a significant advantage in finding these issues.

Why HTTP Knowledge is Critical for Bug Hunters

  • Understanding how data flows helps identify injection points
  • Headers often contain sensitive information or misconfigurations
  • Status codes reveal application behavior and potential information leakage
  • Request/response manipulation is key to testing for vulnerabilities

HTTP Request & Response Flow

1

Client Sends Request

The client (browser, tool, or script) sends an HTTP request to the server containing method, URL, headers, and optionally a body.

Testing Point: This is where you can manipulate requests to test for vulnerabilities like SQLi, XSS, or IDOR.

2

Server Processes Request

The server interprets the request, processes it according to the application logic, and prepares a response.

Testing Point: Pay attention to how the server handles malformed requests or unexpected input.

3

Server Sends Response

The server sends back an HTTP response containing status code, headers, and the requested resource or an error message.

Testing Point: Analyze responses for information disclosure, error messages, or security header misconfigurations.

4

Client Processes Response

The client receives the response and renders the content, executes scripts, or handles the data accordingly.

Testing Point: Check for client-side vulnerabilities like XSS, or analyze how the application handles different response types.

HTTP Request Structure

An HTTP request consists of:

  • Request Line: Method, URL, and HTTP version
  • Headers: Additional information about the request
  • Empty Line: Separates headers from body
  • Body: Optional data sent with the request
Example HTTP Request
GET /api/users/123 HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Accept: application/json Authorization: Bearer abc123... Cookie: session=xyz789; user=admin Content-Type: application/x-www-form-urlencoded

HTTP Request Methods

GET GET

Retrieve data from the server. Should not modify data.

Security Note: Parameters are visible in URL. Check for information disclosure, IDOR, and SSRF vulnerabilities.

POST POST

Submit data to be processed. Often used for form submissions.

Security Note: Check for CSRF vulnerabilities, input validation issues, and business logic flaws.

PUT PUT

Replace target resource with uploaded content.

Security Note: Verify proper authorization checks for resource modification and test for mass assignment.

DELETE DELETE

Remove the target resource.

Security Note: Critical to check authorization and implement proper access controls. Test for IDOR.

Less Common Methods

PATCH

Apply partial modifications to a resource.

Testing Tip: Test for mass assignment vulnerabilities where users can update fields they shouldn't have access to.

OPTIONS

Describe the communication options for the target resource.

Testing Tip: Check for exposed HTTP methods that could be exploited.

HEAD

Same as GET but without response body.

Testing Tip: Useful for checking if a resource exists without downloading its content.

TRACE

Performs a message loop-back test along the path to the target resource.

Security Risk: Can be used in XST (Cross-Site Tracing) attacks. Should be disabled on production servers.

Common HTTP Request Headers

Authorization

Contains credentials for HTTP authentication.

Security Note: Check for weak tokens, JWT issues, or authentication bypass. Test for token leakage.

Cookie

Contains stored HTTP cookies previously sent by the server.

Security Note: Look for session fixation, insecure flags, or cookie manipulation vulnerabilities.

User-Agent

Identifies the client software.

Security Note: Can be manipulated for client-side attacks or to bypass restrictions. Check for User-Agent based SQLi.

Content-Type

Indicates the media type of the request body.

Security Note: Content type confusion can lead to vulnerabilities like XSS or deserialization attacks.

Security-Focused Headers

X-Forwarded-For

Identifies the originating IP address of a client.

Security Risk: Can be spoofed to bypass IP-based restrictions or logging.

Referer

Address of the previous web page from which a link was followed.

Testing Tip: Check if the application properly validates referer headers for CSRF protection.

Origin

Indicates where a request originates from.

Testing Tip: Test CORS misconfigurations by manipulating Origin headers.

HTTP Request Body

The request body contains data sent to the server, commonly in these formats:

Form Data

Key-value pairs, typically from HTML forms.

Example
username=admin&password=secret123&role=user

Testing Tip: Try parameter pollution (same parameter with multiple values) and check how the application handles it.

JSON

Structured data in JavaScript Object Notation.

Example
{ "username": "admin", "password": "secret123", "role": "user" }

Testing Tip: Test for JSON injection, schema validation bypass, and mass assignment vulnerabilities.

XML

Extensible Markup Language for structured data.

Example
<user> <username>admin</username> <password>secret123</password> <role>user</role> </user>

Security Risk: XML endpoints may be vulnerable to XXE (XML External Entity) attacks.

Security Note: Always test different content types and parameter pollution. Many vulnerabilities arise from improper parsing of request bodies. Also test for HTTP Request Smuggling by manipulating Content-Length and Transfer-Encoding headers.

Request Parameters

Parameters can be sent in different parts of an HTTP request:

URL Parameters

Sent in the query string of the URL after the ? character.

Example
GET /search?q=test&category=books&limit=10 HTTP/1.1

Testing Tip: These are visible in logs, browser history, and Referer headers. Check for sensitive data exposure.

Path Parameters

Sent as part of the URL path, often used in REST APIs.

Example
GET /api/users/123/posts/456 HTTP/1.1

Testing Tip: Test for IDOR (Insecure Direct Object Reference) by changing these values.

Body Parameters

Sent in the request body, typically with POST, PUT, or PATCH requests.

Example
POST /api/users HTTP/1.1 Content-Type: application/x-www-form-urlencoded username=admin&password=secret123

Testing Tip: Test for SQL injection, XSS, and command injection in these parameters.

Header Parameters

Sent as HTTP headers, often used for authentication or metadata.

Example
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... X-API-Key: 12345abcde

Testing Tip: Check for hardcoded keys, weak JWT implementations, and header injection vulnerabilities.

HTTP Response Structure

An HTTP response consists of:

  • Status Line: HTTP version, status code, and status text
  • Headers: Metadata about the response
  • Empty Line: Separates headers from body
  • Body: The requested resource or message
Example HTTP Response
HTTP/1.1 200 OK Content-Type: application/json Content-Length: 45 Set-Cookie: session=abc123; Secure; HttpOnly X-Frame-Options: DENY Strict-Transport-Security: max-age=31536000 {"status": "success", "data": "user info"}

HTTP Status Codes

200 OK

Standard successful response.

Check for sensitive data in successful responses.

201 Created

Resource created successfully.

Verify the created resource has proper access controls.

204 No Content

Success but no content to return.

Even with no content, check headers for information leakage.

301 Moved Permanently

Resource has a new permanent URL.

Check if redirects can be exploited for open redirect vulnerabilities.

302 Found

Temporary redirection.

Test for open redirects by manipulating the Location header.

400 Bad Request

Malformed request.

Error messages might reveal application details or vulnerabilities.

401 Unauthorized

Authentication required.

Check for authentication bypass techniques.

403 Forbidden

Server refuses to respond.

Try different HTTP methods or paths to bypass restrictions.

404 Not Found

Resource not found.

Check for information disclosure in error messages.

500 Internal Server Error

Generic server error.

Error messages might reveal backend technology or configuration details.

502 Bad Gateway

Invalid response from upstream server.

Could indicate infrastructure issues or misconfigurations.

503 Service Unavailable

Server temporarily unavailable.

Might be caused by resource exhaustion attacks.

Bug Hunter Tip: Pay close attention to error responses. They often reveal information about the application structure, technologies used, or potential attack vectors. Also test how the application handles unexpected status codes.

Important HTTP Response Headers

Set-Cookie

Sets cookies in the client's browser.

Security Note: Check for missing Secure, HttpOnly, or SameSite flags which can lead to session hijacking.

Content-Security-Policy

Defines allowed sources for content to prevent XSS.

Security Note: Weak or missing CSP can allow XSS attacks. Test for bypasses using unsafe-inline or data: URIs.

Strict-Transport-Security

Forces browsers to use HTTPS.

Security Note: Missing HSTS headers can allow downgrade attacks. Check for proper max-age and includeSubDomains directives.

X-Content-Type-Options

Prevents MIME type sniffing.

Security Note: Missing this header can lead to content sniffing attacks. Should be set to "nosniff".

Additional Security Headers

X-Frame-Options

Prevents clickjacking by controlling iframe embedding.

Testing Tip: Should be set to DENY or SAMEORIGIN. Missing this header allows clickjacking attacks.

Referrer-Policy

Controls how much referrer information is included in requests.

Testing Tip: Prevents leakage of sensitive URL parameters to external sites.

Feature-Policy

Allows control over which browser features can be used.

Testing Tip: Restricting features like camera, microphone, or geolocation enhances privacy.

HTTP Response Body

The response body contains the actual data returned by the server. This can be HTML, JSON, XML, or any other format.

HTML Responses

Typically returned for web pages.

Example
<html> <head> <title>User Dashboard</title> </head> <body> <h1>Welcome, admin</h1> <div>Your user ID is 123</div> </body> </html>

Testing Tip: Check for sensitive data in HTML comments, hidden fields, or JavaScript variables.

JSON Responses

Common in APIs and modern web applications.

Example
{ "status": "success", "data": { "user": { "id": 123, "username": "admin", "email": "admin@example.com", "role": "administrator" } } }

Testing Tip: Look for information disclosure in JSON responses, especially in error messages.

XML Responses

Used in SOAP APIs and some legacy systems.

Example
<?xml version="1.0" encoding="UTF-8"?> <response> <status>success</status> <user> <id>123</id> <username>admin</username> <email>admin@example.com</email> </user> </response>

Security Risk: Check for XXE vulnerabilities in XML responses.

Bug Hunter Tip: Always examine response bodies for information disclosure, even in error messages. Also check for inconsistent responses that might indicate application logic flaws.

Bug Hunting Tips & Common Vulnerabilities

Injection Attacks

Occur when untrusted data is sent to an interpreter as part of a command or query.

Testing Approach: Send special characters, SQL commands, or script tags in parameters and observe responses for errors or unexpected behavior. Test for SQLi, NoSQLi, OS command injection, and LDAP injection.

Tools: SQLmap, NoSQLmap, Commix

Broken Authentication

Flaws in authentication mechanisms allow attackers to compromise passwords or session tokens.

Testing Approach: Test for weak password policies, session fixation, credential stuffing, or lack of rate limiting on login endpoints. Check for exposed session tokens in URLs or logs.

Tools: Burp Suite, OWASP ZAP, custom scripts

Sensitive Data Exposure

Applications may not properly protect sensitive data like passwords or personal information.

Testing Approach: Check if data is transmitted over unencrypted channels, if passwords are hashed properly, or if sensitive data is exposed in responses. Look for PII, credentials, or API keys in client-side code.

Tools: Burp Suite, browser dev tools, grep

XML External Entities (XXE)

Poorly configured XML processors evaluate external entity references within XML documents.

Testing Approach: Send XML with external entity references to endpoints that process XML and observe if they're resolved. Try to read local files or initiate SSRF attacks.

Tools: XXE Injector, custom payloads

Broken Access Control

Restrictions on what authenticated users can do are not properly enforced.

Testing Approach: Try to access resources belonging to other users, escalate privileges, or bypass authorization checks by manipulating request parameters. Test for IDOR, privilege escalation, and CORS misconfigurations.

Tools: Burp Suite, Autorize extension

Security Misconfigurations

Insecure configurations in any part of the application stack.

Testing Approach: Check for default credentials, unnecessary services, verbose error messages, missing security headers, or exposed administrative interfaces.

Tools: Nmap, Nikto, security headers scanners

Cross-Site Scripting (XSS)

Allows attackers to inject client-side scripts into web pages viewed by other users.

Testing Approach: Inject script payloads into all user-input areas and observe if they're executed. Test for reflected, stored, and DOM-based XSS.

Tools: XSS Hunter, BeEF, manual testing

Cross-Site Request Forgery (CSRF)

Tricks users into executing unwanted actions in a web application they're authenticated to.

Testing Approach: Create malicious pages that submit requests to the target application and check if they're executed without user interaction. Test state-changing operations.

Tools: CSRF PoC generators, Burp Suite

Server-Side Request Forgery (SSRF)

Allows attackers to make requests from the server to internal or external resources.

Testing Approach: Manipulate URL parameters to make the server request internal resources or external domains you control. Test for access to metadata services in cloud environments.

Tools: Burp Collaborator, SSRF testing servers

Advanced Bug Hunting Techniques

HTTP Request Smuggling

Exploits differences in how servers parse HTTP requests to bypass security controls.

Technique: Send requests with conflicting Content-Length and Transfer-Encoding headers to poison the request queue.

Test with tools like HTTP Request Smuggler or manually craft requests with TE.CL or CL.TE vulnerabilities.

Race Conditions

Exploits timing vulnerabilities in concurrent processing.

Technique: Send multiple requests simultaneously to exploit limits, balances, or resource allocation.

Use tools like Turbo Intruder in Burp Suite or write custom scripts to send concurrent requests.

API Testing

Focuses on finding vulnerabilities in API endpoints.

Technique: Test for mass assignment, broken object level authorization, and excessive data exposure.

Use tools like Postman, Insomnia, or Burp Suite with API-specific extensions.

Business Logic Flaws

Exploits flaws in application workflow rather than technical vulnerabilities.

Technique: Manipulate application flow to bypass intended restrictions or exploit pricing, inventory, or workflow logic.

Requires deep understanding of application functionality. Manual testing is most effective.
